# Dédalo v7 admin areas & the maintenance widget framework

An **area** is an ontology *model* that is a navigational/structural container, not a data table: it has a `tipo` but **no `section_id` and no `matrix_*` row**. All areas extend `area_common` (→ `common`). One area — `area_maintenance` (tipo `dd88`, `DEDALO_AREA_MAINTENANCE_TIPO`) — is the admin dashboard, and it is the **only** area with a **widget** system: collapsible panels under `core/area_maintenance/widgets/<id>/` that run admin actions (backup, config, migration, diffusion, integrity, system, dev). "Admin area" generally = the `area_*` family; "widgets" = the maintenance-only panel framework. This file is the map and the non-negotiable rules.

## Hard rules

1. **The registry is the source of truth, NOT the `widgets/` directory.** `area_maintenance::get_ar_widget_ids()` (class.area_maintenance.php:478) lists the **29** dispatchable widgets; `get_ar_widgets()` (:129) builds the same set as full descriptors for the UI. There are **31** dirs on disk — `php_user` and `regenerate_relations` exist but are **unregistered → not dispatchable** (`widget_request` whitelists against `get_ar_widget_ids()` at dd_area_maintenance_api:286). A widget that isn't in both `get_ar_widgets()` AND `get_ar_widget_ids()` does not exist as far as the API is concerned. **Adding a widget = create the dir AND register it in BOTH methods** (keep them in the same order).

2. **`get_ar_widgets()` and `get_ar_widget_ids()` MUST stay in sync.** The drift guard `area_maintenance_Test::test_get_ar_widget_ids_matches_get_ar_widgets()` fails if they diverge. The split exists so the API can validate widget names cheaply (no eager DB/network side effects) on every poll, while `get_ar_widgets()` is allowed expensive build-time probes (publication_api → diffusion map, sequences_status → DB, php_user → system) — those eager calls are why `background:true` widgets (`update_data_version`, `diffusion_server_control`, `system_info`) defer their load on the JS side.

3. **Two dispatch entry points, two allowlists — never bypass either.** All maintenance API traffic goes through `dd_area_maintenance_api` (registered in `dd_manager` $allowed_api_classes; access gated to global-admin OR developer with permission ≥ 2). Its own `API_ACTIONS` (:49) is the outer "is this method even an API action?" gate. Then:
   - **`class_request`** (:95) → a `public static` method on `area_maintenance`, gated by **`area_maintenance::API_ACTIONS`** (SEC-044). For area-wide actions (`set_maintenance_mode`, `set_notification`, `rebuild_lang_files`…).
   - **`widget_request`** (:254) → a `public static` method on a widget class, gated by that **widget's own `API_ACTIONS`** (SEC-044). For per-widget actions.
   - **`get_widget_value`** (:439) → read-only `<widget>::get_value()` with **no arguments** (SEC-050 bare-identifier regex + realpath), for polling/refresh.

4. **Widget dispatch is defence-in-depth — keep all four gates.** `widget_request` resolves the widget by: (1) `sanitize_key_dir($source->model)`, (2) whitelist against `get_ar_widget_ids()`, (3) **SEC-069 realpath confinement** — the resolved `widgets/<id>/class.<id>.php` must `str_starts_with(realpath(widgets_root)+SEP)`, (4) `method_exists`, then (5) the per-widget `API_ACTIONS` allowlist. Back-compat: a widget WITHOUT an `API_ACTIONS` constant falls back to "any public-static method is callable" — so **every new widget MUST declare `API_ACTIONS`** or it silently exposes all its statics.

5. **Widgets are static, stateless classes.** No constructor, no instance state. Contract: `public static function get_value() : object` returning `{result, msg, errors}` (read-only probe, called by `get_widget_value` with no args), plus any action methods `set_*/rebuild_*/…` taking a single `object $options` and returning `{result, msg, errors, …}`. Declare `public const API_ACTIONS = [...]` listing every remotely-callable method (always include `get_value`).

6. **`get_value` must never throw and never block hard.** It runs at page build (or idle for `background` widgets). External probes (diffusion engine, DB, filesystem) must fail soft into the `errors[]`/result payload — e.g. media_control probes the engine via `diffusion_api_client::call` which never throws. A throwing `get_value` breaks the whole dashboard render.

7. **Config writes from the UI go through `set_config_core()` and land in `../private/state.php` — NOT `config_core.php`.** `area_maintenance::set_config_core()` (:1121, `protected static`) is the single write-gate: root-gated (`logged_user_id() !== DEDALO_SUPERUSER` rejects, the lone exception is `DEDALO_RECOVERY_MODE` so a locked-out box can recover), per-constant type validation, then it writes the v7 STATE store (`dirname(DEDALO_CONFIG_PATH,2).'/private/state.php'`) via `installer_config_persistor::render_state()` with `LOCK_EX`. The v7 config flip quarantined `config_core.php` out of the web root — never reintroduce a `define()` write there. Allowlisted constants (the `switch`, :1148) are the ONLY settable ones: `DEDALO_MAINTENANCE_MODE_CUSTOM`, `DEDALO_RECOVERY_MODE`, `DEDALO_MEDIA_ACCESS_MODE_CUSTOM`, `DEDALO_NOTIFICATION_CUSTOM` (XSS-sanitized via `safe_xss`); `default:` rejects. To add a settable constant: add a `case` AND a `state_path_map` entry. See [[config-read-emitted-constants]] / [[config-php-not-bootstrap]].

8. **Background execution is its own allowlist.** A `class_request` with `options.background_running:true` is handed to `exec_::request_cli()` → detached `process_runner.php`, which only runs methods in `area_maintenance::BACKGROUND_RUNNABLE` (:62 — `update_data_version`, `long_process_stream`). Response becomes `{result, pid, pfile, msg}`, not the method's return. Synchronous-only widget actions are intentionally absent. The JS side sets `background_running` only where the widget descriptor had `background:true`.

9. **An area is an ontology model with no row identity.** `area_common` (extends `common`, area_common:38) is the base for all areas; `get_section_id()` always returns `null`, `get_section_tipo()` returns the area's own tipo (duck-type shim for section-shaped callers). Areas are resolved by model name via `ontology_utils::get_ar_tipo_by_model('area_maintenance')`; the tipo constant (`DEDALO_AREA_MAINTENANCE_TIPO='dd88'`) is registered in `core/area/class.area.php`. Most `area_*` classes (`area_admin`, `area_resource`, `area_tool`, `area_activity`, `area_publication`, `area_root`, `area_development`) are **intentional stubs** — empty bodies that exist only so the model-registry can locate a tipo; all behaviour is inherited from `area_common`. Don't add logic to a stub unless that area genuinely needs it.

10. **Widgets belong to `area_maintenance` only.** `get_ar_widgets()` exists nowhere else; no other `area_*` dir has a `widgets/` subdir. `area_thesaurus`/`area_ontology` have the **tree** widget (a different system — see [[dedalo-ts-tree]]); the dashboard areas render the `area_common` dashboard (`get_dashboard_data()`, section list + `metric_total` + `metric_activity_30d`). Don't generalise the maintenance widget framework into the base without a real second consumer.

## Architecture map

- **Base** `core/area_common/class.area_common.php` (`extends common`): `get_instance(model, tipo, mode='list')` factory, `get_json()` dispatch (delegates to a concrete `<area>_json.php` if present, else `area_common_json.php`), `get_dashboard_data()` / `get_dashboard_child_sections()` / `count_section_records()` / `metric_total()` / `metric_activity_30d()` (direct SQL on `matrix_activity` dd542) / static `get_activity_metric()`. Custom `_json.php` exist for: `area_maintenance`, `area_thesaurus`, `area_ontology`, `area_graph`; all other areas fall back to `area_common_json.php`.
- **Areas on disk** (`core/`): `area_common`(base), `area`, `area_root`, `area_admin`, `area_activity`, `area_resource`, `area_tool`, `area_thesaurus`, `area_ontology`(extends area_thesaurus), `area_graph`, `area_publication`, `area_development`, `area_maintenance`.
- **Maintenance class** `core/area_maintenance/class.area_maintenance.php` (`extends area_common`): `get_ar_widgets()` (:129), `get_ar_widget_ids()` (:478), `widget_factory()` (:544), `set_config_core()` (:1121) + thin setters `set_maintenance_mode`/`set_media_access_mode`/`set_recovery_mode`/`set_notification`, `get_file_constants()` (:588, SEC-069 realpath-confined to DEDALO_CONFIG_PATH/CORE_PATH), `get_definitions_files()` (:1488), `update_data_version`, `update_ontology`, `rebuild_lang_files`, `register_tools`. Constants: `API_ACTIONS` (:85), `BACKGROUND_RUNNABLE` (:62).
- **API** `core/api/v1/common/class.dd_area_maintenance_api.php` (`final`): `API_ACTIONS` (:49), `class_request` (:95), `widget_request` (:254), `get_widget_value` (:439), plus `lock_components_actions`, `modify_counter`, `get_simple_schema_changes_files`, `parse_simple_schema_changes_files`.
- **`widget_factory()` output shape** (every field always present): `{id, class, category, type:'widget', tipo, parent, label, info, body, run:[], trigger, value, background:false}`. `category` ∈ `data|config|migration|diffusion|integrity|system|dev|general` (drives sidebar grouping + display order). `value` lets the server pre-load a payload so JS skips an extra fetch; `background:true` defers the load to idle.
- **Widget folder** `core/area_maintenance/widgets/<id>/`: `class.<id>.php` (static logic + `API_ACTIONS`), `js/<id>.js` (ES module: `init`→`build`→`render` lifecycle + action methods that call the API), often `js/render_<id>.js`, `css/<id>.less`.
- **Client** `core/area_maintenance/js/area_maintenance.js` (lifecycle, prototypes from `common` + `render_area_maintenance`) + `js/render_area_maintenance.js` (`get_content_data` builds sticky toolbar + per-category `.maintenance_group` grids; `get_category_defs` orders categories; lazy-imports each widget module). Styling: `core/area_maintenance/css/area_maintenance.less` (shell) + per-widget `widgets/<id>/css/<id>.less`.
- **Permissions**: maintenance access is hard-gated in `core/security/class.security.php` (`tipo===DEDALO_AREA_MAINTENANCE_TIPO` ⇒ global-admin OR developer, else 0); `dd_manager` requires permission ≥ 2 before routing to the API; config writes additionally require `DEDALO_SUPERUSER` inside `set_config_core()`.

## Adding a maintenance widget (checklist)

1. `mkdir core/area_maintenance/widgets/<id>/` with `class.<id>.php` (class `<id>`, all methods `public static`), `js/<id>.js`, `css/<id>.less`.
2. In `class.<id>.php`: declare `public const API_ACTIONS = ['get_value', ...]` (rule 4 — omit it and ALL statics become callable). Implement `get_value():object` returning `{result,msg,errors}`, failing soft (rule 6).
3. Register in **both** `area_maintenance::get_ar_widgets()` (full descriptor via `widget_factory`) **and** `get_ar_widget_ids()`, same position (rules 1–2).
4. If it persists config: route through `set_config_core()` with a new `switch` case + `state_path_map` entry; do NOT write `config_core.php` (rule 7).
5. If it runs long: set `background:true` in the descriptor AND, only if it dispatches via `class_request` with `background_running:true`, add the method to `area_maintenance::BACKGROUND_RUNNABLE` (rule 8).
6. Import the widget LESS alongside the others; add a `*_widget_Test.php`.

## Test architecture (keep it green)

- **Area**: `test/server/area/area_maintenance_Test.php` — `test_get_ar_widgets()`, the **drift guard** `test_get_ar_widget_ids_matches_get_ar_widgets()`, `test_widget_factory()`. New widget ⇒ this drift guard catches an unsynced pair.
- **Per widget**: `test/server/area/<id>_widget_Test.php` (e.g. `media_control_widget_Test.php`, `diffusion_server_control_widget_Test.php`). Pattern: exercise `get_value()` and the VALIDATION (no-write) paths of setters. **TEST_USER_ID is DEDALO_SUPERUSER**, so never drive a setter down its valid write path in tests — it would write the real `../private/state.php`; assert on type/validation rejections and force external probes to a dead endpoint for graceful-failure coverage (see [[dedalo-media-protection]] for the media_control example).
- **API**: dispatch-gate tests for `class_request`/`widget_request`/`get_widget_value` (SEC-044/050/069, API_ACTIONS allowlist, realpath confinement).
- Run: `cd test/server && ../../vendor/bin/phpunit area/area_maintenance_Test.php area/<id>_widget_Test.php`. `php -l` every touched PHP file; keep phpstan at 0 on the area + widget. Full DB-gated runs need the dev flag (see [[run-full-server-suite-dev-flag]]).

## Gotchas

- Editing only the `widgets/` dir without touching `get_ar_widget_ids()` ⇒ a 404-style "not a valid widget" from `widget_request`, even though the files exist. The whitelist, not the filesystem, decides.
- `get_widget_value` calls `get_value()` with **no arguments** — never rely on `$options` there; use `widget_request` for anything that needs input.
- `set_config_core()` is `protected static`: it's reached only via the thin public setters listed in `API_ACTIONS`, never dispatched directly. Adding a setter means adding BOTH the public method to `area_maintenance::API_ACTIONS` and the constant case to `set_config_core()`.
- On a fresh/unmigrated box `../private/state.php` may not exist yet — `set_config_core()` tolerates that (`$existing=[]`); don't assume the file is present.
- Don't confuse the maintenance **widget** with the thesaurus/ontology **tree** widget — different subsystem ([[dedalo-ts-tree]]).
